## ① 非エンジニア向けのざっくり概要

別のシステムへ頼みごとをするときの窓口。

プログラム同士がデータを渡したり処理を依頼したりするときの決まりです。Web では HTTP とセットで語られることが多いです。

補足（比喩）：レストランの「メニュー」と「厨房へのオーダー方法」に近いです。

## ② ちょっとだけ詳しく

API は、システム同士が決まった形式で依頼と返答をやり取りする仕組みです。画面を直接操作しなくても、データ取得や登録を自動で連携できます。

動き方は「1) 呼び出し側が URL と方法（GET/POST など）を指定して依頼 -> 2) 必要なデータや認証情報を送る -> 3) 受け側が処理して結果を返す -> 4) 呼び出し側が結果やエラー内容に応じて次の処理を決める」です。

典型シーンは、受発注システムと在庫システムを自動連携するときです。手作業での転記を減らし、更新遅れを防ぐために使われます。

混同しやすいのは画面連携です。API は「機械同士の連携」、GUI は「人の操作」で、連携の主体が違います。

## ③ もっと知りたい人向け

### アーキ設計
公開APIと内部APIを分離し、認証境界と責務を明確にします。REST/gRPC/Webhookの使い分けを、遅延要件と開発体制に合わせて決めるのが基本です。

### 運用設計
バージョン管理、互換性ポリシー、認可スコープ、レート制限、監査ログを運用ルール化します。OpenAPIを起点に仕様・テスト・ドキュメントを同時更新すると品質を保ちやすくなります。

### 障害/変更時の論点
詰まりやすいのは、破壊的変更の告知不足、トークン期限切れ連鎖、依存先障害でのタイムアウト連鎖です。フォールバック応答と段階的リリースを設計しておく必要があります。

### これで実際にできること
- システム間連携を自動化し、手作業連携を減らせる
- 仕様変更の影響範囲を把握し、障害を未然に防げる
- 利用ログを分析して、性能改善や不正検知に活かせる
