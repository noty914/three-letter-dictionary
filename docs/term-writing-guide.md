# 用語執筆ガイド（①/②/③）

このガイドは、`src/content/terms/*.md` の本文品質をそろえるための執筆基準です。

## 1. 読者レベル定義

- ① 非エンジニア向けのざっくり概要: 略語を初めて見た人
- ② ちょっとだけ詳しく: 定義はわかるが、実務での使い方が曖昧な人
- ③ もっと知りたい人向け: さらに深掘りしたい人（公式情報へ案内）

## 2. レベルごとの書き方

### ① 非エンジニア向けのざっくり概要

書く順番:

1. 一言（20字前後）
2. 軽い説明（1〜2段落）
3. 任意で比喩（1段落）

ルール:

- 最初の1文で意味がわかるようにする
- つまずき論点や例外運用は書かない
- 専門語で専門語を説明しない
- 「結論が後ろ」にならないようにする
- カタカナ語は、より平易な日本語に言い換えられるなら言い換える（例: スイート -> サービス一式）

### ② ちょっとだけ詳しく

書く順番:

1. Why｜何のために使うか
2. Who｜誰（どんな役職・スキルセットの人）が主に使うか
3. When｜どんな場面で使うか
4. Where｜どこで動く/使うか（社内・クラウド・端末など）
5. How｜どうやって使うか
6. できること｜導入後の実務効果（3点）
7. よくある誤解｜似た用語との差分（1点）

ルール:

- What は初級（① 非エンジニア向けの概要）と重複するため割愛する
- 2〜5段落で収める
- 実務の会話で出る単語を優先する
- 仕様の細部は上級へ回す
- 専門語を使う場合は、同じ文で短い言い換えを添える
- 読後に「自分の言葉で説明できる」構造にする
- How は 2〜4 ステップで書く

### ③ もっと知りたい人向け

書く順番:

1. 公式サイト/公式仕様へのリンク（1〜3件）
2. 必要なら補足リンク（公的機関・標準化団体など）

ルール:

- まず公式情報を優先して掲載する
- リンク先の役割（公式ドキュメント/仕様/管理者ヘルプ）を短く添える
- 1用語あたり最大3リンク程度に抑える

## 4. 公開前チェック

①:

- 1文目だけ読んで意味がわかるか
- 難語の連鎖が起きていないか

②:

- Why/Who/When/Where/How が埋まっているか
- How が2〜4ステップで説明できるか
- 「できること」が3項目あるか
- 「よくある誤解」が1項目あるか

③:

- 公式リンクが1件以上あるか
- リンク先の役割がわかるか
- リンク数が多すぎないか（目安3件）

frontmatter:

- `related` が `related: [a, b, c]` 形式（小文字・重複なし）になっているか
- `related` の各リンクが相互参照（双方向）になっているか

## 5. 運用ルール（推奨）

- 執筆順は必ず「① -> ② -> ③」
- 1回の作業単位は5〜8語
- バッチ単位でレビューし、都度全件見直しはしない
- `related` は必ず双方向にする（A が B を指すなら、B も A を指す）
- `related` の件数は 2〜5件を目安にする（0件や過剰リンクを避ける）
- 用語追加・更新後は `npm run sync:related` を実行して双方向リンクを自動補正する
